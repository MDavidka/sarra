AI Session

Control per-project Syte cloud runtime conversations for long-lived coding sessions. Independent from preview/deploy processes.

AI session usage. One embedded Syte cloud runtime with durable per-project sessions. Use Syte chat and activity endpoints for normal work; agent_proxy_url is reserved for authenticated native conversation calls. Model profiles: syra-nano (Vertex AI Gemini 3.1 Flash Lite), syra-base (DeepSeek V4 Flash), syra-havy / pro (Vertex AI Gemini 3.6 Flash), syra-ultra (Aliyun Qwen 3.6 / qwen3.5-flash). Each profile thinks and builds. Configure Vertex AI + DeepSeek + Aliyun API keys per profile in Syte GUI → AI. Internal sycord.com routes: /api/internal/projects/{uuid}/agent/* with X-Syra-Internal-Secret.

GET /agent_session/{session_id}?since_id=0
Fetch a durable agent session (Turso access route)
This replaces the old SSE activity stream. Every event produced while the agent works a turn — the request, its plan, tool calls, and the final reply — is written to a durable Turso (libSQL) session identified by a UUID as it happens. Instead of holding open a streaming connection, fetch the whole session document by its id. Asking the agent something is unchanged: agent_communicate/agent_change return this session's id in turso_session_id.
Query parameters
since_id?int
Only return events with a strictly greater id. Useful for polling a session whose status is still open without re-fetching everything already seen.
{
  "ok": true,
  "id": "b6f2b6b6c2e94e2e9e3e4b6c2e94e2e9",
  "project_id": "my-site-a1b2c3",
  "session_number": 3,
  "model_profile": "syra-base",
  "status": "completed",
  "created_at": "2026-07-15T12:00:00+00:00",
  "updated_at": "2026-07-15T12:00:04+00:00",
  "events": [
    {"id": 1, "event_type": "request_started", "role": "user", "detail": "Add dark mode toggle"},
    {"id": 2, "event_type": "processing", "role": "system", "detail": "Cloud agent accepted the durable request"},
    {"id": 3, "event_type": "thinking", "role": "assistant", "detail": "Inspect the current theme first"},
    {"id": 4, "event_type": "tool_call_started", "payload": {"tool": "write_file", "arguments": {"path": "src/theme.ts"}}},
    {"id": 5, "event_type": "tool_call_finished", "payload": {"tool": "write_file", "ok": true}},
    {"id": 6, "event_type": "request_completed", "payload": {"reply": "Added dark mode"}}
  ]
}
Turn lifecycle (correlate events by payload.request_id):
request_started        role=user     the submitted message
  → processing         accepted, durable request queued
  → thinking           (optional) plan / reasoning — also persisted in agent_plans
  → question           (optional) interactive ask_question / request_env widget
  → screenshot         (optional) desktop + phone preview captures
  → tool_call_started  ┐ repeated 0+ times, one pair per tool call
  → tool_call_finished ┘
  → request_completed | request_failed | agent_stopped   (terminal)
Event payloads. request_started/processing/thinking carry payload.request_id. tool_call_started carries payload.tool and payload.arguments; tool_call_finished carries payload.tool and payload.ok (boolean). screenshot carries payload.screenshots[] (viewport, image_url, optional chat_image_base64). question carries question_id, question_type (answer|input|slider|choice|multi_choice), and widget fields. request_completed carries payload.reply; request_failed carries payload.error; agent_stopped carries stopped_at + reason.
status values: open while the turn is in progress, then completed, failed, cancelled, or stopped. Poll GET /api/agent_session/{id}?since_id=<highest event.id seen> until status != "open". Answer pending question events with POST /agent_answer_question so the turn can continue.
Requires Turso to be configured (turso_database_url, optional turso_auth_token, saved from the GUI's AI tab) — returns HTTP 503 otherwise.
async function pollAgentSession(sessionId) {
  let sinceId = 0;
  for (;;) {
    const res = await fetch(`/api/agent_session/${sessionId}?since_id=${sinceId}`, {
      headers: { 'x-api-key': API_KEY },
    });
    const session = await res.json();
    for (const event of session.events) {
      sinceId = Math.max(sinceId, event.id);
      renderEvent(event);
    }
    if (session.status !== 'open') return session;
    await new Promise(r => setTimeout(r, 2000));
  }
}

400 agent_start_failedSyte cloud runtime missing or configuration invalid — install the project Python dependencies
400 backend_unreachableProvider API unreachable — check AI tab → model API keys

sycord.com change request flow

1User edits project on sycord.com and sends a change message
2sycord.com prewarms with POST /api/agent_warm; the supervisor keeps the runtime alive
3Open SSE GET /api/projects/{uuid}/agent/activity/stream?live=1&since_id=N (optionally &format=tagged)
4sycord.com → POST /sycord/api/agent_change or POST /api/internal/projects/{uuid}/agent/change — returns request_id immediately
5SSE streams planning, tool starts/results, token deltas, and exactly one terminal event
6Preview hot-reloads edited files; deploy when user is ready
Sycord public API: GET /sycord/api/agent_status, POST /sycord/api/agent_change, GET /sycord/api/agent_activity — same API token as other /sycord/api/* routes.
Metrics: DPFA = Dedicated Performance For Agents (CPU). MNOA = Maximum Number Of Agents (concurrent active cloud sessions).
